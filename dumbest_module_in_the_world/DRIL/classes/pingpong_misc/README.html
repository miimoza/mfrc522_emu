Le but

Le but est de modéliser un périphérique en mode caractère, qui permettent la lecture et l’écriture de façon simple: quand on envoie "ping", il répond "pong" à la prochaine lecture…​et c’est tout.

Nous avons déjà fait cet exercice dans l’introduction aux devices en mode caractère. Ici, le but est de réimplémenter la même logique, mais en exploitant la classe préfaite "misc", qui devrait nous simplifier la vie.

Un point important: nous n’allons enregistrer qu’un seul misc device, ce qui veut dire que le driver ne pourra gérer qu’un seul périphérique virtuel. Cela aussi, ça nous simplifie la vie…​

Le code

Enregistrement

On doit naturellement toujours enregistrer un device en mode caractère. Sauf que cette fois-ci, cdev_add() et ses amies sont l’affaire de miscdevice, ce qui simplifie agréablement le code des fonctions init() et exit().

La structure miscdevice a plusieurs champs, mais je ne touche qu’à quelques-uns d’entre eux:

  • minor: Indispensable. C’est la même logique que l’allocation statique ou dynamique de nombre majeur, que nous connaissons déjà. Ici, une macro magique, MISC_DYNAMIC_MINOR, permet d’en obtenir un à la volée, ce qui est la méthode à favoriser.

  • name: Purement pour nous, puisqu’il s’agit du nom du noeud qu’on trouvera dans le devfs, et aussi celui du device dans le sysfs. Ne l’oubliez pas: sinon, votre code va échouer de façon bizarre (sur mon PC, je me suis pris un ENOMEM…​).

  • fops: On peut passer nos propres implémentations de fonctions, sachant que la classe misc implémente déjà quelques fonctions, comme open() par exemple. En pratique, si vous donnez votre open(), elle appellera la sienne, puis la vôtre. Les curieux pourront lire misc_open() dans drivers/char/misc.c, qui est bien commentée.

Pourquoi ne pas toucher aux autres champs ?

  • Certains champs, comme list, sont initialisés par misc_register() quoiqu’on fasse.

  • D’autres, comme groups (i.e. les groupes d’attributs) sont valables si nuls; dans ce cas précis, on dit juste qu’on ne veut pas ajouter d’autres attributs.

Après l’enregistrement

On note deux évènements après avoir inséré notre module:

  • Un noeud a été automatiquement créé dans le devfs. Si vous avez appelé votre device "toto", alors /dev/toto sera là, avec le majeur 10 (misc) et le mineur que vous avez alloué.

  • Votre device est visible sous la classe misc dans le sysfs; notre ami "toto" se trouve dans /sys/class/misc/toto, et il a des attributs de base assez mystérieux:

    #> tree /sys/class/misc/ppm

Le propos de ces attributs n’est pas nécessairement intéressant pour le cours, c’est plutôt la mécanique qui permet d’en créer des similaires qui va nous intéresser.

Read & Write

Les deux fonctions de lecture et d’écriture de notre "pingpong" initial sont reprenables telles quelles, ou presque. Comme vous pouvez le constater dans le code, j’ai pris la liberté d’utiliser un booléen global pour savoir si nous avons été "pingués", et non plus une variable propre à chaque device.
